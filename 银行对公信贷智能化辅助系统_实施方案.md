# 实施方案更新记录

## 更新：程序化文档生成 + 资料上传重构 + 财务分析升级（v5.0）✅ 已完成

### 更新背景

v4.x 的"模板填空"方案存在根本缺陷：XML Run 层级占位符替换脆弱、模板字段名与 AI 输出不匹配、模板结构（341段落+25表格）过于复杂、Phase3 的 10 次 Minimax API 调用与 Phase2.5 的 42 字段推理内容重叠冗余。

v5.0 完全舍弃 Word 模板，改用 python-docx 从零构建文档，同时重构资料上传清单、升级财务分析能力。

---

### 一、核心架构变更

```
Phase1（MinerU 解析）
  └─ 多元化资料上传（重构）
        │
        ▼
Phase2（数据提取 + AI 推理）
  ├─ 2.1 财务报表提取（资产负债表/利润表/现金流量表 + 合并/非合并识别）
  ├─ 2.2 科技创新特征提取
  ├─ 2.3 企业基本信息提取
  ├─ 2.4 财务指标计算（杜邦分析 + 11 项年度指标）
  ├─ 2.5 收入交叉核验（报表营收 vs 银行流水 vs 纳税申报）
  ├─ 2.6 AI 推理（42 字段）
  └─ ❌ 删除合规筛查
        │
        ▼
Phase3（文档生成）
  └─ DocxReportBuilder：python-docx 从零构建 .docx
```

**关键变化**：
- 舍弃 Word 模板 → python-docx 从零构建
- 删除 MinimaxReportGenerator（10 次 API 调用）：Phase2.5 推理已覆盖全部 10 章内容
- 删除 ReportTemplateFiller：不再需要 XML Run 级占位符替换
- 删除合规信息核实

### 三、Phase2 财务分析升级

#### 3.1 财务报表提取字段

**资产负债表**：
总资产、总负债、所有者权益、流动资产、流动负债、应收账款、应付账款、存货、货币资金、交易性金融资产、应收票据、短期借款、应付票据、长期借款、应付债券

**利润表**：
营业收入、营业成本、利润总额、净利润

**现金流量表**：
经营性净现金流、投资性净现金流、融资性净现金流

**报表类型识别**：通过关键词（"合并"、"合并报表"）自动识别合并/非合并报表，合并报表优先用于分析

#### 3.2 收入交叉核验

三方数据源交叉验证：

| 数据源 | 来源 | 提取内容 |
|--------|------|----------|
| 财务报表 | B1 | 营业收入（利润表） |
| 银行流水 | B2 | 贷方发生额合计 |
| 纳税申报表 | B3 | 申报营业收入 |

核验逻辑：
- 报表 vs 银行流水：偏差率 ≤10% 基本吻合，10%~30% 存在偏差，>30% 重大偏差
- 报表 vs 纳税申报：偏差率 ≤5% 基本一致，5%~15% 存在差异，>15% 重大差异
- 数据不足时自动标注缺失项

#### 3.3 财务指标计算（11 项年度指标）

| 指标 | 公式 |
|------|------|
| 资产负债率 | 总负债 / 总资产 × 100% |
| 流动比率 | 流动资产 / 流动负债 |
| 速动比率 | (流动资产 - 存货) / 流动负债 |
| 应收账款周转天数 | 360 / (营业收入 / 应收账款) |
| 应付账款周转天数 | 360 / (营业成本 / 应付账款) |
| 存货周转天数 | 360 / (营业成本 / 存货) |
| 销售利润率 | 利润总额 / 营业收入 × 100% |
| 毛利率 | (营业收入 - 营业成本) / 营业收入 × 100% |
| 刚性负债 | 短期借款 + 应付票据 + 长期借款 + 应付债券 + … |
| 现金类资产 | 货币资金 + 交易性金融资产 + 应收票据 |
| 刚性负债净敞口 | 刚性负债 − 现金类资产 |

**杜邦分析**：ROE = 净利率 × 资产周转率 × 权益乘数

#### 3.4 删除合规筛查

- `_run_phase2()` 进度从 5 步变为 6 步（新增收入交叉核验，删除合规）
- `run_inference()` 移除 `compliance_data` 参数
- `phase2_compliance.py` 不再被导入

## 更新：Human-in-the-Loop + Excel 分流强化（v3.2）

### 更新背景

v3.1 的人工核对环节放在 Phase 1（查看原始 Markdown），存在以下问题：
1. **原始 Markdown 可读性差**：OCR 解析后的表格/数字未结构化，信贷员难以核对
2. **LLM 幻觉风险**：直接用原始 Markdown 生成报告，可能放大 OCR 错误
3. **Excel 分流不够严格**：上传阶段未对 Excel 文件做强制路由

v3.2 将人工核对移至 Phase 2 之后，以结构化表格形式展示财务指标，信贷员可直接修正 AI 提取的数值后再生成报告。

```

    ↓

    ↓
★★★ Human-in-the-Loop：核对结构化数据（st.data_editor）★★★
    ↓
Phase3 Gemini 报告生成
```

## 更新：AI 推理后置 + 并行批量优化（v5.1）✅ 已完成
